These options are specific to mold and have no equivalent in GNU ld. For GNU-compatible options, see [GNU-compatible options](/reference/gnu-compatible-options). For `-z` flags, see [-z options](/reference/z-options).

## --chroot

```
--chroot=dir
```

Set `dir` as the root directory.

## --color-diagnostics

```
--color-diagnostics=[auto|always|never]
--color-diagnostics
--no-color-diagnostics
```

Show diagnostic messages in color using ANSI escape sequences.

- `auto` — print in color only if stderr is connected to a TTY. This is the default.
- `always` — always use color.
- `never` — never use color.

`--color-diagnostics` (without a value) is a synonym for `--color-diagnostics=auto`.
`--no-color-diagnostics` is a synonym for `--color-diagnostics=never`.

## --detach / --no-detach

```
--detach
--no-detach
```

Permit or do not permit mold to create a debug info file in the background.

When `--separate-debug-file` is used, mold creates the debug info file in the background by default (`--detach`). Pass `--no-detach` if you need the debug file to be complete before mold exits.

## --fork / --no-fork

```
--fork
--no-fork
```

Spawn a child process and let it do the actual linking. When linking a large program, the OS kernel can take a few hundred milliseconds to terminate a mold process. `--fork` hides that latency by having the parent return immediately. By default, mold does fork.

<Note>
`--fork` also hides the resource usage statistics reported by `time(2)`, since the parent does not call `waitpid(2)` on the child process. Pass `--no-fork` if you need those statistics.
</Note>

## --perf

```
--perf
```

Print performance statistics after linking completes.

## --print-dependencies

```
--print-dependencies
```

Print dependency information for input files. Each line of output shows which file depends on which file to use a specific symbol.

This option is useful for debugging why an object file in a static archive was linked, or why a shared library is kept in the output file's dependency list even with `--as-needed`.

## --relocatable-merge-sections

```
--relocatable-merge-sections
```

By default, mold does not merge input sections by name when combining input object files into a single output object file for `-r`. For example, `.text.foo` and `.text.bar` are not merged even though they would normally be merged into `.text`.

This option changes that behavior so that mold merges input sections by name according to the default section merging rules.

## --repro

```
--repro
```

Archive input files, along with a text file containing the command-line options, into a tar file so that you can reproduce the exact same link invocation. This is useful for filing bug reports with a reproducer.

The output tar file is named `path/to/output.tar`, where `path/to/output` is the output filename specified by `-o`.

You can also enable this with the `MOLD_REPRO` environment variable.

## --reverse-sections

```
--reverse-sections
```

Reverse the order of input sections before assigning them offsets in the output file.

This is useful for finding bugs that depend on the initialization order of global objects. In C++, constructors of global objects within a single source file are guaranteed to run in source order, but there is no such guarantee across compilation units. Reversing the section order lets you easily test that your program works correctly with a reversed initialization order.

## --run

```
--run command arg...
```

Run `command` with mold as `/usr/bin/ld`. Specifically, mold sets the `LD_PRELOAD` environment variable to intercept `exec(3)` family functions and replaces `argv[0]` with itself if it is `ld`, `ld.gold`, or `ld.lld`.

```bash
mold --run gcc -o hello hello.c
```

## --separate-debug-file

```
--separate-debug-file[=file]
--no-separate-debug-file
```

Bundle debug info sections into a separate file instead of embedding them in the output executable or shared library. By default, mold creates the debug info file in the background so that you can start running your executable as soon as possible.

The debug info file is created in the same directory as the output file, with the `.dbg` extension. The filename is embedded in the output so that GDB can automatically locate it. For more information, see [GDB separate debug files](https://sourceware.org/gdb/current/onlinedocs/gdb.html/Separate-Debug-Files.html).

mold uses `flock(2)` to hold a file lock while creating the debug info file in the background.

To suppress background creation, pass `--no-detach`. To disable the feature entirely, use `--no-separate-debug-file`.

## --shuffle-sections

```
--shuffle-sections
--shuffle-sections=number
```

Randomize the output by shuffling the order of input sections before assigning offsets. If a seed `number` is given, it is used for the random number generator so that the linker produces the same output for the same seed. If no seed is given, a random number is used.

This option is useful for benchmarking. Modern CPUs are sensitive to a program's memory layout — a seemingly benign layout change can affect performance. By running benchmarks with randomized memory layouts, you can isolate real performance changes from noise caused by layout differences.

```bash
# Reproducible shuffle with seed 42
mold --shuffle-sections=42 -o hello hello.o

# Random shuffle each time
mold --shuffle-sections -o hello hello.o
```

## --spare-program-headers

```
--spare-program-headers=number
```

Append the given number of `PT_NULL` entries to the end of the program header. This allows post-link processing tools to add new segments by overwriting the null entries.

<Note>
ELF requires all `PT_LOAD` segments to be sorted by `p_vaddr`. If you add a new `LOAD` segment, you may need to sort the entire program header.
</Note>

## --stats

```
--stats
```

Print input statistics after linking.

## --thread-count

```
--thread-count=count
```

Use `count` threads for linking.

## --threads / --no-threads

```
--threads
--no-threads
```

Use multiple threads. By default, mold uses as many threads as the number of CPU cores, capped at 32 (mold does not scale well beyond that point).

To use only one thread, pass `--no-threads` or `--thread-count=1`.

## --quick-exit / --no-quick-exit

```
--quick-exit
--no-quick-exit
```

Use or do not use `quick_exit(3)` to exit. `quick_exit` is the default.

## --zero-to-bss

```
--zero-to-bss
```

Convert all-zero data sections into BSS.

When enabled, mold scans input data sections that are not of type `SHT_NOBITS` and checks whether their contents consist solely of zero bytes. Such sections are converted into BSS (`SHT_NOBITS`) sections, reducing the size of the output file since BSS sections occupy no space in the file image.

This is especially useful for user-defined sections created with `__attribute__((section(".sectname")))` that contain uninitialized global variables. GCC and Clang do not automatically mark such sections as BSS even if their contents are entirely zero.

```c
// Without --zero-to-bss: emitted as SHT_PROGBITS filled with zeros
// With --zero-to-bss: converted to SHT_NOBITS (smaller output file)
__attribute__((section(".mysection"))) int vec[256];
```
